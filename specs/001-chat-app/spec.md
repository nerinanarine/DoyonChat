# Feature Specification: OpenCode Go Chat Web App

**Feature Branch**: `[001-opencode-chat]`

**Created**: 2026-06-06

**Status**: Draft

**Input**: User description: "生成AIと対話型でチャットできるWebアプリ。OpenCode Goを使用。Reactフロントエンド。Azureデプロイ。MVP: テキストチャット、ストリーミング応答、会話履歴保持、複数会話管理、Markdown/コードブロックレンダリング、画像入力、レスポンシブデザイン。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - テキストチャットとストリーミング応答 (Priority: P1)

ユーザーがテキストメッセージを入力し、送信ボタンまたはEnterキーで送信すると、AIからの応答がストリーミング（逐次表示）で受信され、リアルタイムに画面に表示される。

**Why this priority**: チャットアプリの最も核心的な機能。なければプロダクトとして成立しない。

**Independent Test**: フロントエンドのみで、バックエンドAPIをモック化して、メッセージ送信とストリーミング表示が動作することを確認できる。

**Acceptance Scenarios**:

1. **Given** チャット入力欄にテキストが入力されている状態、**When** ユーザーが送信ボタンをクリックまたはEnterを押す、**Then** メッセージがチャット履歴に追加され、AI応答がストリーミングで表示される
2. **Given** AIが応答を生成中の状態、**When** ユーザーが画面を見ている、**Then** 応答が文字単位またはトークン単位でリアルタイムに追加表示される
3. **Given** ストリーミング応答中の状態、**When** ユーザーが新しいメッセージを送信する、**Then** 前のストリーミングは停止し、新しいメッセージの送信が開始される

---

### User Story 2 - 複数会話の管理と履歴保持 (Priority: P1)

ユーザーは複数の会話（チャットスレッド）を作成・管理でき、各会話の履歴はデータベースに永続化され、ページリロード後も復元される。サイドバーで会話を一覧表示し、切り替え・削除ができる。

**Why this priority**: 実用的なチャットアプリに必須。単一の会話だけでは継続的な利用が困難。

**Independent Test**: バックエンドAPIを経由して会話の作成、一覧取得、メッセージ追加、会話切り替えが動作することを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーが新しい会話を開始したい状態、**When** 「新規チャット」ボタンをクリックする、**Then** 新しい空の会話が作成され、サイドバーに追加される
2. **Given** 複数の会話が存在する状態、**When** ユーザーがサイドバーの会話をクリックする、**Then** その会話のメッセージ履歴が表示され、そこから継続してチャットできる
3. **Given** ユーザーが会話を削除したい状態、**When** 削除アイコンをクリックする、**Then** 確認後、その会話が削除され、サイドバーから消える
4. **Given** ユーザーがページをリロードした状態、**When** ページが読み込まれる、**Then** 以前の会話一覧と各会話のメッセージが復元される

---

### User Story 3 - Markdownとコードブロックのレンダリング (Priority: P2)

AIの応答に含まれるMarkdown記法（見出し、リスト、リンク、テーブルなど）とコードブロックが、適切にスタイル付けされたHTMLとしてレンダリングされる。コードブロックにはシンタックスハイライトとコピーボタンが付く。

**Why this priority**: チャットアプリの品質に直結。AIの応答を読みやすくするため必須。

**Independent Test**: 静的なMarkdownテキストをフロントエンドに渡して、正しくレンダリングされ、コードブロックにコピーボタンが表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** AIの応答にMarkdown記法が含まれている状態、**When** 応答が表示される、**Then** 見出し、リスト、リンク、テーブルなどが適切にスタイル付けされて表示される
2. **Given** AIの応答にコードブロックが含まれている状態、**When** 応答が表示される、**Then** シンタックスハイライトが適用され、コピーボタンが表示される
3. **Given** ユーザーがコピーボタンをクリックした状態、**When** コピーアクションが実行される、**Then** コードブロックの内容がクリップボードにコピーされる

---

### User Story 4 - 画像入力（マルチモーダル）(Priority: P2)

ユーザーはチャット入力時に画像をアップロードでき、画像付きのメッセージを送信できる。AIは画像を理解した上でテキスト応答を返す。

**Why this priority**: OpenCode Goのモデル（GLM-5.1, Kimi K2.6等）が画像入力に対応しており、現代の生成AIチャットの標準機能。

**Independent Test**: フロントエンドで画像をアップロードし、OpenCode Go APIに正しい形式（Base64エンコードされたimage_url）で送信されることを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーが画像をアップロードしたい状態、**When** 画像ファイルをドラッグ＆ドロップまたはファイル選択でアップロードする、**Then** 画像のプレビューが入力欄に表示される
2. **Given** 画像付きのメッセージを送信する状態、**When** 送信ボタンをクリックする、**Then** 画像がBase64エンコードされ、OpenAI互換の`image_url`形式でAPIに送信される
3. **Given** AIが画像を受信した状態、**When** AIが応答を生成する、**Then** 画像の内容に基づいたテキスト応答が返される

---

### User Story 5 - レスポンシブデザイン（モバイル対応）(Priority: P2)

アプリはデスクトップ、タブレット、モバイルの各画面サイズで適切に表示され、操作性が確保される。モバイルではサイドバーがハンバーガーメニュー化され、チャット画面が全幅で表示される。

**Why this priority**: 現代のWebアプリの標準要件。Azure上で公開するアプリとしては必須。

**Independent Test**: ブラウザの開発者ツールで各画面サイズをシミュレートし、レイアウトが崩れないことを確認できる。

**Acceptance Scenarios**:

1. **Given** デスクトップ画面（1280px以上）でアプリを開いた状態、**When** ページが表示される、**Then** サイドバーとチャット画面が左右に並んで表示される
2. **Given** モバイル画面（768px以下）でアプリを開いた状態、**When** ページが表示される、**Then** サイドバーが非表示になり、ハンバーガーメニューから開閉できる
3. **Given** タブレット画面（768px〜1024px）でアプリを開いた状態、**When** ページが表示される、**Then** 適切なレイアウトで表示され、タッチ操作も可能

---

### User Story 6 - モデル切り替え (Priority: P2)

ユーザーは会話ごとにOpenCode Goで利用可能なAIモデルを選択・切り替えできる。チャット画面のヘッダーまたは入力欄近くにモデル選択UIがあり、デフォルトモデル（kimi-k2.6）から他のモデルに変更できる。モデル選択は会話単位で保存され、切り替え後のメッセージは新しいモデルで生成される。

**Why this priority**: 異なるモデルは異なる特性（品質、速度、コスト、マルチモーダル対応など）を持つ。ユーザーが用途に応じてモデルを選べることで、使い勝手が大幅に向上する。

**Independent Test**: モデル選択UIからモデルを変更し、APIリクエストに正しいmodelパラメータが含まれることを確認できる。

**Acceptance Scenarios**:

1. **Given** チャット画面が表示されている状態、**When** ユーザーがモデル選択ドロップダウンを開く、**Then** OpenCode Goで利用可能なモデル一覧が表示される
2. **Given** モデル選択ドロップダウンが開いている状態、**When** ユーザーが別のモデルを選択する、**Then** 選択されたモデルがハイライトされ、以降のメッセージはそのモデルで生成される
3. **Given** モデルを切り替えた会話が存在する状態、**When** ページをリロードする、**Then** 選択されたモデル設定が復元され、継続してそのモデルでチャットできる
4. **Given** 画像をアップロードした状態、**When** マルチモーダル非対応モデルを選択する、**Then** 画像付きメッセージ送信時に警告が表示される

---

### Edge Cases

- **OpenCode Go APIエラー時**: タイムアウト、レート制限、無効なAPIキーなどのエラーに対して、ユーザーに分かりやすいエラーメッセージを表示する
- **ストリーミング中断時**: ネットワーク切断やブラウザリロード時に、途中までの応答を履歴に保存する
- **長い会話履歴**: 会話が長くなりすぎた場合（トークン数超過）、自動的に要約やスレッド分割を検討する（MVPでは警告表示のみ）
- **画像サイズ制限**: 大きすぎる画像はアップロード前に圧縮または警告を表示する（上限5MB程度）
- **同時アクセス**: 同一ユーザーが複数タブで開いた場合、最新の状態が同期される（MVPでは手動リロードで同期）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ユーザーはテキストメッセージを入力して送信でき、AIからのストリーミング応答をリアルタイムに表示できる
- **FR-002**: システムは会話履歴をデータベースに永続化し、ページリロード後も復元できる
- **FR-003**: ユーザーは複数の会話を作成・管理でき、サイドバーで一覧表示と切り替えができる
- **FR-004**: ユーザーは会話を削除できる
- **FR-005**: AIの応答に含まれるMarkdownを適切にレンダリングし、コードブロックにはシンタックスハイライトとコピーボタンを付ける
- **FR-006**: ユーザーは画像をアップロードしてマルチモーダルメッセージを送信できる
- **FR-007**: システムはOpenCode Go API（OpenAI互換Chat Completions）を使用してAI応答を生成する
- **FR-008**: APIキーはバックエンドで管理し、フロントエンドからは隠蔽する
- **FR-009**: フロントエンドはデスクトップ、タブレット、モバイルの各画面サイズで適切に表示される
- **FR-010**: ストリーミング中に「停止」ボタンを表示し、ユーザーが応答生成を中断できる
- **FR-011**: ユーザーは会話ごとに利用するAIモデルを選択・切り替えできる
- **FR-012**: システムは会話に紐づくモデル設定を永続化し、復元できる
- **FR-013**: モデル選択UIには、各モデルの特性（品質、速度、マルチモーダル対応など）を表示する
- **FR-014**: マルチモーダル非対応モデル選択時に画像付きメッセージを送信しようとした場合、警告を表示する

### Key Entities

- **Conversation（会話）**: チャットスレッドを表す。ID、タイトル、使用モデル、作成日時、更新日時、メッセージリストを持つ
- **Message（メッセージ）**: 単一のチャットメッセージを表す。ID、会話ID、役割（user/assistant）、内容、画像URL（オプション）、作成日時を持つ
- **Model（モデル）**: OpenCode Goで利用可能なAIモデル。ID、表示名、説明、マルチモーダル対応フラグ、推奨用途を持つ
- **User（ユーザー）**: MVPでは匿名ユーザー。セッションまたはローカルストレージで識別

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがメッセージを送信してから、AI応答の最初の文字が表示されるまでの時間（Time To First Token）が3秒以内
- **SC-002**: ページリロード後、会話履歴とメッセージが正しく復元される
- **SC-003**: モバイル（iPhone SE〜iPhone 15 Pro Max相当の画面サイズ）でチャット送信から応答表示までの操作が完了できる
- **SC-004**: コードブロックのコピーボタンが、クリック後1秒以内にクリップボードにコピーする
- **SC-005**: OpenCode Go APIエラー時、ユーザーに分かりやすいエラーメッセージが5秒以内に表示される
- **SC-006**: モデル切り替えは2クリック以内で完了し、切り替え後の最初のメッセージが新しいモデルで生成される
- **SC-007**: モデル一覧の表示が1秒以内に完了する

## Assumptions

- ユーザーはモダンなブラウザ（Chrome, Firefox, Safari, Edgeの最新版）を使用する
- OpenCode Go APIはOpenAI互換のChat Completions API（SSEストリーミング）を提供する
- Azure上でバックエンド（Node.js）とフロントエンド（React）を分離してホスティングする
- MVPではユーザー認証は不要（匿名ユーザー、セッションまたはローカルストレージで識別）
- 画像アップロードはBase64エンコードで直接送信し、別途ストレージサービスは使用しない（MVPでは5MB以下の画像のみ）
- ダークモードはMVPから除外（Phase 2で追加検討）
